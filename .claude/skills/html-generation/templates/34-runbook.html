P1 Ingestion lag exceeds 5 minutes INC-001
Symptoms
  • Alert: meridian.ingestion.lag_seconds > 300 firing in Grafana
  • Dataset status stuck in "indexing" for > 10 minutes
  • Kafka consumer group meridian-transform offset not advancing
Diagnosis
  1. 1

    Check Kafka consumer lag across all partitions:

    $ kafka-consumer-groups.sh \
    --bootstrap-server kafka:9092 \
    --group meridian-transform \
    --describe
GROUP              TOPIC         PARTITION  CURRENT-OFFSET  LOG-END  LAG
meridian-transform raw-events    0          824491          825301   810
meridian-transform raw-events    1          812042          812042   0
  2. 2

    Check Transform Engine pod status (lag on partition 0 above suggests a stuck pod):

    $ kubectl get pods -n meridian-processing -l app=transform-engine
$ kubectl describe pod transform-engine-xxxxx -n meridian-processing
    Look for
    OOMKilled, CrashLoopBackOff, or ImagePullBackOff in the Events section
  3. 3

    Tail logs for the stuck pod to confirm root cause:

    $ kubectl logs -n meridian-processing \
    deploy/transform-engine --tail=200 -f
Do not delete the stuck pod before capturing logs. Log evidence is required for the post-mortem.
Remediation
  1. A

    If OOMKilled: increase memory limits and restart:

    $ kubectl set resources deploy/transform-engine \
    -n meridian-processing \
    --limits=memory=4Gi
$ kubectl rollout restart deploy/transform-engine -n meridian-processing
  2. B

    If CrashLoopBackOff (code bug): roll back to previous image:

    $ kubectl rollout undo deploy/transform-engine -n meridian-processing
$ kubectl rollout status deploy/transform-engine -n meridian-processing
  3. C

    Verify lag is clearing post-restart (allow 3 minutes):

    $ watch -n 15 kafka-consumer-groups.sh \
    --bootstrap-server kafka:9092 \
    --group meridian-transform \
    --describe | grep LAG
    Expected
    LAG column should decrease to 0 on all partitions within 5 minutes
Escalation path
0 – 15 min: On-call engineer self-resolves using this runbook

Page via PagerDuty if alert fires

15 – 30 min: Escalate to Platform Lead

Slack @platform-lead • Phone if unresponsive

> 30 min unresolved: Declare P1 incident

P2 Query API timeout spike (> 5% error rate) INC-002
Symptoms
  • HTTP 504 or 408 responses visible in Grafana API Error Rate panel
  • P99 query latency > 3 seconds on meridian.query.latency_p99
Diagnosis
  1. 1

    Check Redis cache hit rate — a drop to < 60% indicates cache eviction:

    $ redis-cli -h redis.meridian.internal INFO stats \
    | grep -E "keyspace_hits|keyspace_misses"
  2. 2

    Check columnar index health on the query executor pods:

    $ kubectl exec -n meridian-serving \
    deploy/query-executor \
    -- curl -s http://localhost:9090/healthz/index
    Expected
    {"status":"ok","index_lag_ms":0,"replicas":3}
Remediation
  1. A

    If cache eviction: flush stale keys and expand Redis memory:

    $ redis-cli -h redis.meridian.internal \
    FLUSHDB ASYNC
# Then update memory-policy in values.yaml and redeploy
  2. B

    If index replica unhealthy: force re-sync from Object Store:

    $ kubectl exec -n meridian-serving \
    deploy/query-executor \
    -- curl -X POST http://localhost:9090/admin/resync-index
P3 Index build stuck in "indexing" > 30 min INC-004
Quick fix
  1. A

    Trigger a manual index re-run via the admin API (does not disrupt active queries):

    $ curl -X POST https://api.meridian.io/v2/admin/datasets/{id}/reindex \
    -H "Authorization: Bearer $ADMIN_API_KEY"
  2. B

    If re-index fails within 10 minutes, check the Index Builder logs:

    $ kubectl logs -n meridian-processing \
    deploy/index-builder --tail=300 | grep ERROR
