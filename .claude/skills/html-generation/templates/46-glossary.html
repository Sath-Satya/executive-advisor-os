SponAI Consulting — Glossary

Standard terminology used across SponAI Consulting engagements, deliverables, and internal documentation.

A 3 terms
ADRCore
Architecture Decision Record. A short document capturing a significant architectural decision, its context, the options considered, the chosen option, and the rationale. Stored in memory/projects/{domain}/{project}/decisions.md.
AgentforceSF
Salesforce's native AI agent platform. Enables configuration of autonomous agents that can reason, plan, and execute actions within a Salesforce org using Einstein AI and grounding data from Data Cloud.
Agentic OSAI
SponAI's internal meta-OS framework managing five domain-specific operating systems (Salesforce, .NET, Python, iOS, TradingSystems). Each domain OS is a self-contained git repo with synced skills, agents, commands, and memory.
B 2 terms
BlazorCore
Microsoft's framework for building interactive web UIs using C# instead of JavaScript, running on WebAssembly (Blazor WASM) or server-side (Blazor Server). SponAI's default frontend for .NET projects.
BlockerCore
A dependency or impediment that prevents a gate from advancing. Documented in memory/projects/{domain}/{project}/blockers.md. Requires explicit resolution before the pipeline can continue.
C 4 terms
CQRSCore
Command Query Responsibility Segregation. An architectural pattern separating read models (queries) from write models (commands). Used in SponAI's .NET projects with MediatR for clean handler dispatch.
Claude CodeAI
Anthropic's CLI-based coding agent. SponAI uses Claude Code as the primary implementation engine across all five domain OSes, with skills, agents, hooks, and commands defined per domain.
Container AppInfra
Azure Container Apps — SponAI's default deployment target for Python and .NET microservices. Serverless, scale-to-zero, with built-in KEDA scaling and managed certificates.
Conventional CommitsCore
A commit message convention used across all SponAI repos. Format: type(scope): description. Types: feat, fix, docs, refactor, test.
D 2 terms
Data CloudSF
Salesforce's unified data platform for ingesting, harmonizing, and activating customer data. Used as the grounding source for Agentforce agents in SponAI's healthcare client deployments.
DEV-IDCore
A unique developer identifier used by the session-identity skill to scope per-developer memory files (session handovers, coaching notes) within a shared repo.
E 2 terms
EasyLanguageCore
TradeStation's proprietary scripting language for writing trading strategies, indicators, and functions. Requires explicit begin/end blocks on all if statements.
EF CoreCore
Entity Framework Core. Microsoft's ORM for .NET. SponAI uses EF Core with SQLite for POC and PostgreSQL for production, code-first migrations, and repository pattern layering.
F 2 terms
FLSSF
Field-Level Security. Salesforce permission system controlling read/write access to individual fields. Every Apex data access must check FLS (via WITH SECURITY_ENFORCED or Security.stripInaccessible) to prevent privilege escalation.
FastAPICore
SponAI's default Python web framework for building REST APIs. Auto-generates OpenAPI docs, enforces Pydantic schemas, and supports async/await natively. Paired with Azure Container Apps for deployment.
G 2 terms
GateCore
One stage in SponAI's 7-gate delivery pipeline: G1 Plan, G2 Design, G3 Build, G4 Test, G5 Review, G6 Deploy, G7 Verify. Each gate has defined entry criteria, exit checks, and a responsible agent.
Governor LimitSF
Salesforce multi-tenant runtime constraints (e.g., 150 DML operations per transaction, 100 SOQL queries). All Apex code must be bulkified to avoid hitting governor limits in production.
I 1 term
ICPCore
Ideal Customer Profile. SponAI's primary ICP: mid-market companies (100–5,000 employees), healthcare orgs, and Salesforce-heavy organizations needing custom development, AI integration, or cloud migration.
L 2 terms
LWCSF
Lightning Web Component. Salesforce's modern component model based on native Web Components and ES6+. SponAI uses LWC for all new Salesforce UI development (Aura is legacy).
_learnings.mdAI
The per-skill feedback log file. Entries are written during session wrap-up when a skill produces a surprising or notable result. When 3+ entries accumulate for a pattern, /evolve-skills is triggered to propose a SKILL.md update.
M 2 terms
MVVMCore
Model-View-ViewModel. SponAI's default iOS architecture pattern. ViewModel owns state and business logic; View is purely declarative SwiftUI; Model is the domain layer. Combine or async/await for data binding.
Meta-OSAI
The root orchestration layer at c:/AllAboutAI/OSofOS/. Holds master copies of all Shared-tier skills, agents, commands, and rules. Domain OSes are synced from the Meta-OS via /sync-shared.
P 2 terms
POCCore
Proof of Concept. A time-boxed, scope-limited build demonstrating a specific capability. SponAI POCs are production-ready (Dockerized, CI/CD, tested) — not throwaway prototypes.
Pipeline OrchestratorAI
The SponAI skill that classifies work items, routes them through the 7-gate pipeline, loads appropriate memory layers per gate, and manages gate handovers.
S 3 terms
Semantic KernelCore
Microsoft's SDK for integrating AI models into .NET applications. SponAI uses Semantic Kernel for orchestrating LLM calls, plugin execution, and memory management in .NET-based AI apps.
SOQLSF
Salesforce Object Query Language. SQL-like query syntax for querying Salesforce records. Must never be placed inside loops (governor limit violation). Use collections and bulk patterns instead.
sync-sharedAI
The Meta-OS command and shell script that pushes Shared-tier assets from master to all 5 domain repos using SHA256 drift detection. Master is always the source of truth. Run --dry-run before --apply.
T 2 terms
TDDCore
Test-Driven Development. RED (write failing test) → GREEN (minimal code to pass) → REFACTOR (clean up). SponAI's default build methodology for all non-config work. Minimum 80% coverage for production.
TestFlightCore
Apple's beta testing platform. SponAI iOS projects deploy to TestFlight before App Store submission, allowing client stakeholders to test on real devices without an App Store review cycle.
