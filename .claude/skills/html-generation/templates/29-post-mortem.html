Executive Summary

What Happened

At 09:14 PST on {{INCIDENT_DATE}}, the AI inference API began returning HTTP 503 errors to the prior authorization processing queue. The failure originated from an unhandled out-of-memory (OOM) condition in the AI inference container cluster, triggered by an unexpected spike in concurrent requests following a batch retry of 847 queued requests from the previous night's scheduled maintenance window. The retry batch saturated the container pool at a moment when three of twelve containers were already in a degraded state due to a memory leak introduced in the v2.4.1 deployment two days prior. Service was fully restored at 13:37 PST after rolling restart of all containers and emergency autoscaling rule adjustment.

Impact Analysis

Scope of Impact

During the 4h 23m outage window, the AI-assisted PA review workflow was unavailable. Reviewers reverted to the manual evidence gathering process. The primary impacts were:

  • 3,412 PA requests delayed beyond their standard processing commitment. Of these, 214 were approaching the 72-hour regulatory deadline at time of restoration.
  • 2,200 contracted providers experienced delayed authorization decisions, potentially affecting scheduled procedures.
  • $86,000 estimated incremental operational cost from manual processing overtime, contractor augmentation, and provider relations follow-up.
  • No patient safety events were identified as a result of the delay. No regulatory SLA violations were ultimately triggered, as all 214 near-deadline cases were processed within the 72-hour window following restoration.
Root Cause Analysis

Root Cause

The immediate cause of the outage was container OOM termination driven by two concurrent conditions: an unbounded retry batch (847 simultaneous requests) and a pre-existing memory leak in the v2.4.1 tokenizer module that had degraded three containers to a state where their effective memory headroom was 40% below baseline.

Root cause: The maintenance window retry logic lacked a rate-limiting configuration, defaulting to maximum concurrency on retry. This design assumption had never been explicitly tested under conditions where containers were already in a degraded state. The absence of a staged restart verification step in the deployment runbook for v2.4.1 meant that the memory leak was not detected before peak load hours.

Contributing Factors

Contributing Factors

Process
  • No rate-limit configuration on batch retry logic in maintenance mode
  • Deployment runbook did not include a memory leak detection step post-deploy
  • No formal load test was run before promoting v2.4.1 to production
Technology
  • Memory leak in tokenizer module cache — introduced in v2.4.1, not caught in pre-production testing
  • Autoscaling rules did not account for degraded container state in capacity calculations
  • Alerting threshold (error rate >10%) delayed detection by 8 minutes
People
  • On-call engineer was 12 minutes away from end of shift — context handoff was in progress at alert time
  • Retry logic code review did not flag the missing rate-limit configuration
Environment
  • Outage coincided with peak provider submission hour (09:00–11:00 PST)
  • Batch retry volume (847 requests) was 3x the highest previously observed retry load

Post-Mortem Learnings

Key Learnings

What We Learned
  • Batch retry operations must be treated as first-class load events — not as administrative clean-up tasks — and must be subject to the same rate limiting and load testing that governs normal production traffic.
  • Container degradation signals must be surfaced to the autoscaling layer. A container that is degraded but not yet OOM-terminated creates invisible capacity erosion that is invisible to standard metrics until failure occurs.
  • Memory leak detection cannot be deferred to production observation — a mandatory post-deploy soak test window at realistic load is a prerequisite for future releases.
  • Alert thresholds set for normal variance (10% error rate) are not appropriate for AI inference APIs where any sustained error rate above 2% represents customer impact. Thresholds must be calibrated to the service's SLA, not its historical variance.
