Section 01

Goals and Non-Goals

Goals (In Scope)
  • Provide a real-time AI evidence retrieval API for PA reviewers consuming FHIR-formatted clinical data
  • Return a structured evidence bundle within 5 seconds at p95 under production load
  • Support multi-tenant isolation: each health plan's data is logically separated
  • Provide a guideline-matched recommendation with confidence score and source citations
  • Log every inference request with full provenance for HIPAA audit trail compliance
Non-Goals (Explicitly Excluded)
  • Replace clinical reviewer judgment — AI recommendations are advisory only
  • Support real-time streaming (WebSocket) in this phase — REST only
  • Build the reviewer UI — that is owned by the PA workflow team
  • Train or fine-tune the underlying LLM — we use the hosted inference endpoint
  • Handle inbound provider submissions — out of scope for this API
Section 02

Background and Motivation

Clinical reviewers currently spend 41 minutes per PA request gathering evidence from six disparate source systems. This spec defines the API that consolidates that retrieval into a single endpoint, formats the result as a structured evidence bundle, and runs it through an AI model to produce a guideline-anchored summary.

The AI Evidence API will be consumed by the PA Review UI (owned by the workflow team) and must meet latency, availability, and compliance requirements appropriate for a system that directly supports clinical determinations affecting member care.

Dependency This spec assumes the FHIR Data Aggregation Service (spec TECH-2024-041) is deployed and reachable. The AI Evidence API does not own the source data — it queries and transforms FHIR bundles produced upstream.
Section 03

System Architecture

The AI Evidence API is a stateless Python FastAPI service deployed as a containerized workload on Azure Container Apps. It orchestrates calls to the FHIR Aggregation Service, the AI Inference endpoint (Azure OpenAI), and the Audit Log Service, then assembles and returns a structured JSON response.

Section 05

API Endpoints

All endpoints are versioned under /api/v1/ and require JWT bearer authentication. Rate limiting: 60 requests per reviewer per minute.

Table 2. AI Evidence API endpoint inventory
MethodPathDescriptionAuthSLA
POST /api/v1/evidence Generate evidence bundle for a PA request. Primary endpoint. JWT + plan scope p95 < 5s
GET /api/v1/evidence/{request_id} Retrieve a previously generated evidence bundle by its request ID (cached for 30 min). JWT + plan scope p95 < 200ms
GET /api/v1/guidelines List available guideline versions the API can apply. JWT p95 < 100ms
GET /api/v1/health Health check — returns service status and upstream dependency status. None p95 < 50ms
GET /api/v1/audit/{request_id} Retrieve the full audit trail for an evidence request. HIPAA compliance endpoint. JWT + audit scope p95 < 500ms
Section 06

Authentication and Security

All requests require a short-lived JWT issued by the health plan's identity provider (Azure Entra External ID). The JWT must contain a plan_id claim that matches the request body — the API enforces this at the middleware layer to prevent cross-tenant data access.

HIPAA Requirement All API calls that touch ePHI must be logged with the reviewer ID, plan ID, member ID (hashed), and audit token. The audit log write is synchronous — a failed audit log write returns a 500 and does not return the evidence bundle.

TLS 1.2 minimum is enforced at the Azure API Management gateway. Secrets are stored in Azure Key Vault and loaded via Managed Identity — no secrets in environment variables or application configuration files.

Section 07

Testing Strategy

Testing follows the project's TDD-workflow requirements. The AI inference layer is tested against a deterministic mock that returns fixture responses — tests do not call the live Azure OpenAI endpoint.

  • Unit tests — Pydantic model validation, evidence parsing, guideline matching logic. Target: 85% line coverage.
  • Integration tests — FHIR client against a local FHIR R4 test server. Audit log write verification.
  • Contract tests — Pact consumer contracts against the PA Review UI team's expected request/response shapes.
  • Load tests — k6 load test at 1,000 concurrent requests. P95 latency must remain below 5s. Scheduled to run in staging before each production deploy.
Section 08

Observability

All telemetry is emitted to Azure Monitor via OpenTelemetry SDK. Each request generates a trace with spans for FHIR retrieval, cache lookup, AI inference, and audit log write. The following metrics are recorded per request:

  • evidence_api.request.duration_ms — end-to-end latency histogram
  • evidence_api.fhir.retrieval_ms — FHIR upstream latency
  • evidence_api.ai.confidence_score — AI confidence distribution
  • evidence_api.cache.hit_rate — Redis cache efficiency
  • evidence_api.error.rate — error rate by type (upstream, validation, timeout)
Section 09

Rollout Plan

Phase 1 Internal Pilot {{PHASE_1_DATE}}

Deploy to staging. 5 internal reviewers using the new API alongside the legacy manual process. Compare evidence bundle accuracy against manual gold standard. No real-time traffic.

Phase 2 Controlled Rollout {{PHASE_2_DATE}}

Enable for 20% of production PA requests via feature flag. Monitor latency, error rate, and reviewer override rate. Full rollback capability within 5 minutes.

Phase 3 General Availability {{PHASE_3_DATE}}

Enable for 100% of PA requests. Decommission legacy manual evidence-gathering workflow. 30-day hypercare period with elevated monitoring.

Section 11

Open Questions

  1. Guideline version pinning: Should the API always use the latest guideline version, or should the workflow system pass a specific version? Pinning ensures reproducibility; latest ensures current clinical standards. Decision needed from clinical leadership by {{DECISION_DATE}}.
  2. Audit log retention: Legal requires 7 years. Compliance requires 6 years. Using the more restrictive (7 years) for now — confirm with legal counsel.
  3. Confidence threshold for auto-approval: Should requests above 0.95 confidence be auto-approved without reviewer touch? Requires clinical governance review before any automation is implemented. Out of scope for this spec.
